Appendix F: Technical Documentation and Code Repository

F.1 Technical Architecture Documentation

F.1.1 System Architecture Overview

Architecture Pattern: Client-Server with Offline-First Design
- Frontend: Progressive Web Application (PWA)
- Backend: Node.js with Express.js
- Database: SQLite (local) + PostgreSQL (cloud sync)
- Sync Service: Custom conflict resolution middleware
- Authentication: JWT tokens with offline capability

Component Layers:
1. Presentation Layer (Frontend)
   - React.js with progressive enhancement
   - Service workers for offline functionality
   - IndexedDB for local data storage
   - Canvas-based block programming interface

2. Business Logic Layer
   - Computational thinking engine
   - Adaptive scaffolding algorithms
   - Progress tracking and analytics
   - Cultural context management

3. Data Persistence Layer
   - Local SQLite database for offline operation
   - Cloud PostgreSQL for synchronization
   - File system for project assets
   - Cache management system

4. Integration Layer
   - RESTful API endpoints
   - WebSocket connections for real-time features
   - Third-party service integrations
   - Device-specific hardware interfaces

F.1.2 Database Schema Documentation

Local Database Schema (SQLite):
```sql
-- Users Table
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT UNIQUE NOT NULL,
    school_id INTEGER,
    grade_level INTEGER,
    language_preference TEXT DEFAULT 'en',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_sync TIMESTAMP
);

-- Projects Table
CREATE TABLE projects (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER,
    title TEXT NOT NULL,
    content TEXT, -- JSON representation of block program
    level INTEGER, -- 1, 2, or 3
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    modified_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    sync_status TEXT DEFAULT 'pending'
);

-- Progress Table
CREATE TABLE progress (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER,
    competency_id INTEGER,
    score REAL,
    attempts INTEGER,
    last_attempt TIMESTAMP,
    mastery_level INTEGER
);

-- Sync Queue Table
CREATE TABLE sync_queue (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    operation TEXT, -- 'create', 'update', 'delete'
    table_name TEXT,
    record_id INTEGER,
    data TEXT, -- JSON representation
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    retry_count INTEGER DEFAULT 0
);
```

Cloud Database Schema (PostgreSQL):
```sql
-- Similar structure with additional fields for analytics
CREATE TABLE users (
    id SERIAL PRIMARY KEY,

-- Analytics Tables
CREATE TABLE learning_analytics (
    id SERIAL PRIMARY KEY,

    session_id TEXT,
    activity_type TEXT,
    duration_seconds INTEGER,
    completion_rate REAL,
    help_requests INTEGER,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP

```

F.1.4 Offline Synchronization Protocols

Sync Strategy: Operational Transformation with Conflict Resolution

Upload Process:
1. Client bundles local changes since last sync
2. Changes compressed and encrypted
3. Server applies changes in sequence
4. Conflict detection and resolution
5. Confirmation sent to client

Download Process:
1. Server aggregates changes since client last sync
2. Data compressed for efficient transfer
3. Client applies changes with conflict resolution
4. Local database updated
5. Sync status updated

Conflict Resolution Rules:
- Last-write-wins for user profile data
- Manual resolution for project conflicts
- Automatic merge for progress data
- Version history preservation for audit

F.1.5 Security Implementation Details

Authentication Security:
- JWT tokens with 15-minute expiration
- Refresh tokens with 7-day expiration
- Rate limiting on authentication endpoints
- Device fingerprinting for session management

Data Security:
- AES-256 encryption for sensitive data
- HTTPS/TLS 1.3 for all communications
- SQL injection prevention through parameterized queries
- XSS protection through content security policy

Privacy Protection:
- Data minimization principles
- User consent management
- Data retention policies
- Right to deletion implementation
